## 项目概述

**项目名称**: 云集智能文件清理专家
**核心功能**:
- 智能文件扫描与分类（按类型/大小/时间/重复文件）
- 文件类型自定义颜色标记
- 重复文件检测与批量留存策略
- 预设规则包（7种社区成熟清理规则）
- 分析报告（饼状图+统计+建议方案）
- 文件批量删除/移动/导出

**技术架构**:
- GUI框架: CustomTkinter (基于Tkinter的现代化UI库)
- 图标/图像: Pillow
- 系统信息: psutil
- 打包工具: PyInstaller (`--onedir` 模式)

### 🔧 构建脚本 (`build/build.py`)

**构建命令**：
```bash
# 从项目根目录运行
python build/build.py
```

**构建流程**：
1. 从 `dev/app/` 当前代码直接构建
2. PyInstaller 使用 `--onedir` 模式（**必须**，`--onefile` 会导致 Conda DLL 加载失败）
3. 构建产物输出到 `dev/云集智能文件清理专家-vYYYY.MM.DD.HHMM/` 目录（**必须**）
4. PyInstaller 临时文件放在 `build/vYYYY.MM.DD.HHMM/` 下
5. 自动更新 `version_history.json` 和 `dev/ver/version.json`
6. 构建完成后恢复源代码中的版本号为 `DEV`
7. **不删除旧版构建产物**，方便切换测试

## 开发工作流

### ⭐ 标准开发流程

```
1. 日常开发 → 在 dev/app/ 修改和测试
   - 直接修改 main.py 等文件
   - 运行 python dev/app/main.py 测试

2. 构建EXE → 从项目根目录运行构建脚本
   python build/build.py

3. 测试EXE → 运行 dev/云集智能文件清理专家-vYYYY.MM.DD.HHMM/ 下的EXE验证

4. Git提交 → 提交 dev/app/ 目录
   git add dev/app/
   git commit -m "v版本号: 修改描述"
   git push
```

```
4. Git提交并推送
   git add dev/app/ dev/ver/
   git commit -m "v版本号: 稳定版发布"
   git push
```

## 打包和发布

### 📦 Python 环境

本项目推荐使用 **Conda Python** 进行构建：

| 环境 | 路径 | 版本 | 用途 | 可否构建 |
|------|------|------|------|---------|
| Conda Python | `C:\ProgramData\miniconda3\python.exe` | 3.13.x | EXE构建 | ✅ 需DLL收集 |

### ⚠️ 重要：必须使用 `--onedir` 模式

**为什么不能用 `--onefile`？**

Conda Python 的 DLL（如 `libffi-8.dll`、`tcl86t.dll`、`tk86t.dll`）分散在 `Library/bin` 目录下。`--onefile` 模式下：
1. PyInstaller 将所有文件解压到临时目录
2. 内置的 matplotlib runtime hook 在用户 hook 之前执行
3. 此时 `ctypes` 加载 `_ctypes.pyd`，找不到 `libffi-8.dll`
4. 导致 `ImportError: DLL load failed while importing _ctypes`

`--onedir` 模式下，EXE 和所有 DLL 在同一目录，Windows 原生 DLL 搜索机制就能找到依赖。

### 📦 发布包结构（--onedir 模式）

```
dev/
├── 云集智能文件清理专家-vYYYY.MM.DD.HHMM/
│   ├── 云集智能文件清理专家.exe   # 主程序EXE
│   └── _internal/                # PyInstaller 运行时
│       ├── customtkinter/        # CTk 资源
│       ├── PIL/                  # Pillow
│       ├── tcl/                  # Tcl/Tk 数据
│       ├── _ctypes.pyd           # C类型支持
│       ├── _tkinter.pyd          # Tkinter
│       ├── libffi-8.dll          # ctypes依赖
│       ├── tcl86t.dll            # Tcl运行时
│       ├── tk86t.dll             # Tk运行时
│       ├── icon.ico              # 图标
│       ├── icon.png              # 图标
│       └── ...                   # 其他DLL和依赖
└── app/                          # 项目源代码
```

### 📦 构建脚本自动完成

1. 动态生成版本号：`YYYY.MM.DD.HHMM`
2. 将版本号注入 `main.py` 的 `VERSION` 变量
3. 更新 `version_info.txt`（Windows文件属性）
4. 收集 Conda DLL 依赖（`libffi-8.dll`、`tcl86t.dll`、`tk86t.dll` 等）
5. PyInstaller `--onedir` 构建
6. 将构建产物复制到 `dev/云集智能文件清理专家-vYYYY.MM.DD.HHMM/` 目录
7. 恢复源代码版本号为 `DEV`
8. 更新 `version_history.json` 和 `dev/ver/version.json`
9. **不删除旧版构建产物**，方便切换测试

### 多版本 EXE 保留

- `dev/` 下可能同时存在多个版本EXE目录
- `.gitignore` 已忽略 `dev/云集智能文件清理专家-v*/`，不推送到远程

## 常见问题

### Q: EXE 启动报 "DLL load failed while importing _ctypes"
A: 这是 Conda Python + `--onefile` 模式的已知问题。解决方案：
1. **必须使用 `--onedir` 模式** — EXE 和 DLL 在同一目录
2. 确保 `libffi-8.dll` 被收集到 `_internal/` 目录
3. 不要使用 `--onefile` 模式

### Q: EXE 启动报 "DLL load failed while importing _tkinter"
A: 同上，确保 `tcl86t.dll` 和 `tk86t.dll` 被收集，且 Tcl/Tk 数据目录正确打包。

### Q: 如何回滚到旧版本
A: 从 `build/` 的旧版本目录中找到 `build.py` 备份，用相同参数重新构建。或直接运行 `dev/` 下保留的旧版 EXE 目录。

### Q: EXE启动报 ValueError: are not supported arguments
A: CustomTkinter 的 CTkScrollbar 不支持 `scrollbar_color`/`scrollbar_hover_color` 参数，应使用 `button_color`/`button_hover_color`。构建前务必用 `python -c "import main"` 测试代码能否正常加载。
